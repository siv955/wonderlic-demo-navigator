# MIGRATION_NOTES.md

Guidance for another developer or AI assistant taking over the codebase, especially if migrating out of Lovable.

## Stack

- **Framework**: TanStack Start v1 (React 19, SSR-capable). File-based routing under `src/routes/`.
- **Bundler**: Vite 7.
- **Styling**: Tailwind CSS v4 via native `@import "tailwindcss"` in `src/styles.css`. No `tailwind.config.js`.
- **UI**: shadcn/ui ("new-york" style) in `src/components/ui/*`. Icons via lucide-react.
- **State/data**: All content is static. TanStack Query is installed and wired into the router context but the current pages do not fetch anything at runtime.
- **Package manager**: bun (see `bunfig.toml`, `package.json`).
- **TypeScript**: strict mode. Path alias `@/*` → `src/*`.

## How to run locally

```bash
bun install
bun run dev   # http://localhost:8080
bun run build # production build (Nitro / Cloudflare Workers target)
```

`vite.config.ts` extends `@lovable.dev/vite-tanstack-config`, which pre-registers TanStack Start, React, Tailwind, tsconfig-paths, Nitro (Cloudflare target), and error plumbing. Do **not** re-register those plugins manually.

## Data dependencies

None external at runtime. All content lives in `src/data/`. There is no database, no auth, no backend API. If you migrate off Lovable, no Cloud/Supabase migration is needed for this app.

## Lovable-specific patterns to strip if migrating out

- `@lovable.dev/vite-tanstack-config` in `vite.config.ts` — replace with a direct TanStack Start Vite config listing the plugins explicitly.
- `src/lib/lovable-error-reporting.ts`, `src/lib/error-capture.ts`, `src/lib/error-page.ts` — Lovable-specific error surfacing. Safe to remove if you provide your own error boundaries.
- `src/server.ts` — thin SSR error wrapper for the Lovable sandbox. Keep or replace.
- `src/routeTree.gen.ts` — auto-generated by the TanStack Router Vite plugin. Do not hand-edit. It regenerates on `dev`/`build`.
- `.lovable/` metadata folder — remove.
- Hosting target: `@lovable.dev/vite-tanstack-config` defaults to Cloudflare Workers via Nitro. Change the Nitro preset if you target Vercel, Node, etc.

## Known generated / template patterns

- Route files export a `Route` constant from `createFileRoute("<path>")({ head, component })`. The `head()` function returns `{ meta: [...] }` used for SSR-rendered `<title>` / OG tags.
- The archetype dispatch route (`src/routes/archetypes.$id.tsx`) switches on the `$id` path param and returns the matching detail component.
- Every archetype detail component follows the same skeleton: header block → Run This Demo card → Business Impact Ladder → Core Story → Tabs for Select / Develop / Team Dynamics with Accordions inside → Full Close → Recap → Cautions.

## Recommended next technical cleanup

1. Extract a single `<ArchetypeDetail guidance={G} />` component. All six current detail components are near-identical copies; the divergence is only in the guidance object they render. This alone would remove ~4500 lines.
2. Add a `RichGuidance` interface (see `DATA_MODEL.md`) and type all six `*_GUIDANCE` constants against it. Right now they are inferred `as const` objects, so nothing catches missing fields at compile time.
3. Retire `archetypeGuidance.ts`, `productMoments.ts`, and `TALK_TRACKS` once nothing imports them.
4. Add a build-time content check (Zod parse) so a malformed pain, profile, or attribute story fails CI instead of silently rendering wrong.
